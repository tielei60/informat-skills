# Informat 表达式能力边界

本文件用于防止把 JavaScript 逻辑误写成表达式。

## 一、表达式适合做什么

适合：

- 条件判断
- 字符串拼接
- 数值计算
- 日期函数调用
- 数组判断、查找、聚合
- 上下文信息读取

不适合：

- 复杂流程控制
- 多步副作用
- 任意 JavaScript 编程

## 二、这些 JavaScript 写法不要输出

不要输出：

- 箭头函数：`x => x.name`
- 数组高阶方法：`.map(...)`、`.filter(...)`、`.reduce(...)`
- JS 数组字面量：`['a', 'b']`
- 依赖 JS 对象操作的写法

如果用户想要这些能力，应说明“表达式环境不支持这种 JavaScript 写法”。

## 三、优先替代方案

常见替代：

- 数组构造：`Array.of(...)`
- 字符串拼接：`String.concat(...)`
- 条件逻辑：`A ? B : C`
- 集合判断：`Array.containsAny(...)`、`Array.containsAll(...)`
- 聚合：`Array.sum(...)`、`Array.avg(...)`、`Array.max(...)`、`Array.min(...)`
- 日期计算：`Date.dateAdd(...)`、`Date.datePart(...)`、`Date.dateBefore(...)`、`Date.dateAfter(...)`

## 四、日期函数的特殊点

生成日期表达式时，尤其要注意：

- `month` 的值范围是 `0` 到 `11`
- `day_of_week` 的值范围是 `0` 到 `6`

这意味着：

- 1 月不是 `1`，而是 `0`
- 星期日不是 `1`，而是 `0`

如果用户要求“按自然月份数字判断”，要先确认是否需要把用户语义转换为表达式内部取值。

## 五、什么时候建议改用脚本

遇到这些需求时，优先建议改用 `informat-script`：

- 需要基于复杂循环拼装数据
- 需要多次查询再综合判断
- 需要调用表达式文档里不存在的外部能力
- 需要对不确定字段做较复杂的容错处理
